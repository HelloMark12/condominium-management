# MODULE OBJECTIVE

Allow condominium administrators to communicate securely with property owners and tenants inside the application.

Administrators create and manage notices.

Owners and tenants receive a combined, read-only notice feed based on:

- Their active apartment relationships
- The buildings connected to those apartments
- The notice audience
- The notice targets

Owners and tenants cannot reply, edit notices or create official concerns.

# CORE PRINCIPLES

- The condominium administrator remains the manager of all communication.
- One shared app is used by administrators, owners and tenants.
- Permissions are enforced on the backend.
- Archived apartments grant no notice access.
- No resident may see unrelated residents, owners or tenants.
- Administrators may see delivery information only for buildings and apartments managed by their company.
- Owners may see delivery information only for tenants connected to apartments they own.
- Tenants may see only their own read status.
- Notices are never permanently deleted.
- Published-notice edits are permanently auditable.

# NOTICE TARGETING

An administrator must be able to target:

1. All active apartments across the administrator company
2. One building
3. Multiple selected buildings
4. One individual apartment
5. Multiple selected apartments

Targeting rules:

- An apartment target must belong to the administrator’s company.
- A building target must belong to the administrator’s company.
- Archived apartments must not be selectable.
- Inactive or archived buildings must not be selectable.
- The backend must validate every selected building and apartment.
- Do not trust target IDs supplied by the frontend.
- Do not allow targets belonging to another administration company.

If a notice targets both a building and an apartment inside that building, the same recipient should receive the notice only once.

# AUDIENCE OPTIONS

The administrator chooses one audience:

- Owners only
- Tenants only
- Owners and tenants

Audience rules:

Owners only:
- Deliver to users with active ownership relationships for targeted active apartments.

Tenants only:
- Deliver to users with active tenancy relationships for targeted active apartments.

Owners and tenants:
- Deliver to both groups.

A person who owns or rents multiple targeted apartments should see one copy of the same notice in their feed, not duplicates.

The system must still record all relevant delivery relationships internally so administrators can understand which buildings and apartments were covered.

Store categories in a maintainable database enum or equivalent validated structure.

Category labels shown in the UI should be easy to change later.

# NOTICE STATUS

Support:

- Draft
- Scheduled
- Published
- Archived

Rules:

Draft:
- Visible only to authorised administrators.
- Can be edited freely.
- Can be published or scheduled.

Scheduled:
- Visible only to administrators until publication time.
- Can be edited or cancelled before publication.
- Must publish according to Europe/Malta time.

Published:
- Visible to targeted owners and tenants.
- Can be edited, but all changes must be versioned.
- Cannot be converted back into a draft.

Archived:
- Removed from the default active feed.
- Still available in historical notice records.
- Cannot be permanently deleted.

- Targeting mode
- Targeted building IDs
- Targeted apartment IDs
- Status
- Publish immediately option
- Scheduled publication date and time
- Published date and time
- Archived date and time
- Created by administrator

- Updated timestamp

Validation:

- Title is required.
- Body is required.
- At least one valid target is required.
- Audience is required.
- Category is required.
- Scheduled publication must be in the future.
- Use Europe/Malta for scheduling and display.
- Reject invalid or cross-company targets.

# PUBLISHED NOTICE EDITING AND VERSION HISTORY

Published notices may be edited.

Every published edit must:

- Preserve the previous title and body.
- Preserve the previous category, audience and targets.
- Store the administrator who made the change.
- Store the date and time.
- Increment the notice version.
- Store an optional edit reason.
- Never overwrite historical versions.

Create a notice_versions table or equivalent immutable history structure.

Administrators should be able to open a notice and view:

- Current version
- Previous versions
- Date of each edit
- Administrator responsible
- Edit reason where entered

Owners and tenants should see:

- The current version
- An “Updated” label
- The latest update date
- No internal administrator notes

If a published Emergency notice is edited:

- Mark the notice as updated.
- Reset or create a new unread state for the affected recipients so they see the updated emergency message again.
- Preserve their previous read history in the audit data.

For non-emergency published notices:

- Preserve existing read records.
- Show the Updated label.
- Do not automatically make the notice unread again in this first version.

# DELIVERY AND READ TRACKING

Track notice delivery and reads.

The system must know:

- Which user received the notice
- Why they received it
- Relevant apartment relationship
- Relevant building
- User role at the time of delivery
- Delivered timestamp
- First opened timestamp
- Most recent opened timestamp
- Read status
- Notice version last read

Read behaviour:

- Opening a notice marks it as read.
- Reading must be recorded server-side.
- A resident cannot mark another user’s notice as read.
- A user receiving the same notice through multiple apartments sees one feed item.
- Opening that feed item records one user-level read state while preserving internal apartment/building coverage.

Do not expose recipient lists to other residents.

# PUBLISHING AND RECIPIENT GENERATION

When a notice is published:

1. Validate administrator access.
2. Validate status transition.
3. Validate audience and all targets.
4. Resolve active targeted apartments.
5. Resolve eligible active owners and/or tenants.
6. Deduplicate users.
7. Create one notice_delivery per unique user.
8. Create all relevant notice_delivery_context rows.
9. Set published_at.
10. Store the initial immutable notice version.
11. Create building timeline events for every affected building.
12. Commit the complete operation atomically.

Use a database transaction.

If recipient generation fails, the notice must not be partially published.

# SCHEDULED PUBLICATION

Implement scheduled publishing in a maintainable manner.

Use Europe/Malta.

Provide:

- A scheduled publishing service/function
- Idempotent publication
- Safe handling if executed twice
- A method that can be triggered through a scheduled job later
- A lazy fallback check when an administrator accesses the notice area

Automated tests must be able to invoke the scheduled publication service directly.

Do not rely solely on the browser remaining open.
