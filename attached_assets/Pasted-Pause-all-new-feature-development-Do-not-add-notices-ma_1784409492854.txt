# PART 1 — CRITICAL SECURITY FIXES

## 1. Building authorisation

Fix all building routes so users cannot read or modify buildings belonging to another condominium administration company.

Apply backend authorisation to:

* GET /buildings/:buildingId
* PATCH /buildings/:buildingId
* GET /buildings/:buildingId/units

Rules:

Administrator:

* May access a building only when they have an active administrator membership for the building’s company.
* May edit a building only when they are an administrator of that company.

Owner:

* May see limited building information only when they have an active owner relationship with an active apartment in that building.
* May not edit the building.

Tenant:

* May see limited building information only when they have an active tenant relationship with an active apartment in that building.
* May not edit the building.

The backend must:

* Resolve the building from the database.
* Determine the building’s actual company.
* Check the authenticated user’s database relationships.
* Never trust company IDs, building IDs, role claims or route parameters without verification.
* Return 403 or 404 for unauthorised access.
* Never expose another company’s data.

Extract shared building-access helpers where practical so the same logic is not duplicated inconsistently.

---

## 2. Frontend route protection

Add role-aware route guards in App.tsx or the relevant routing layer.

Restrictions:

* /admin/* requires administrator access.
* /owner/* requires at least one active ownership relationship.
* /tenant/* requires at least one active tenancy relationship.

When a user opens an unauthorised route:

* Redirect them to the correct home/dashboard.
* Do not briefly render restricted page content.
* Display an access-denied message where appropriate.

A person may have multiple relationships. For example, they could be an owner of one apartment and a tenant of another. The application should allow them to access the correct sections based on their active relationships.

Frontend route guards are defense-in-depth only. Backend authorisation remains mandatory.

---

## 3. Archived apartment access

When an apartment is archived:

* Preserve the apartment record.
* Preserve ownership history.
* Preserve tenancy history.
* Preserve invitation history.
* Preserve billing history.
* Do not permanently delete records.

However, an archived apartment must stop granting active access.

Owners and tenants must no longer:

* See the apartment in active dashboards.
* Open the apartment through a direct URL.
* Access the apartment through an API request.
* Access the building through that archived relationship.
* Receive active permissions from the archived apartment.

Update all relevant endpoints, including:

* GET /units/:unitId
* GET /auth/me
* GET /me/units
* GET /me/tenancy
* Building-access relationship checks

Owner and tenant access must require both:

1. unit membership status = active; and
2. unit status = active.

The administrator may still view archived apartments and their history.

When an apartment is restored:

* Restore the apartment itself to active status.
* Do not automatically reactivate a previously revoked tenant.
* Do not automatically create new access relationships.
* Existing ownership access may remain subject to the current product rule, but test and document the exact behaviour clearly.

---

## 4. Invitations for archived apartments

Do not allow:

* Owner invitations to be created for archived apartments.
* Tenant invitations to be created for archived apartments.
* Existing pending invitations to be accepted after the apartment is archived.

Return a clear validation error.

---

## 5. Invitation identity verification

An invitation token must not operate as an unrestricted bearer token.

The Clerk-verified email address of the authenticated user accepting the invitation must match the invitedEmail stored against the invitation.

Requirements:

* Compare email addresses case-insensitively.
* Normalise leading/trailing whitespace.
* Use the verified email from Clerk.
* Reject acceptance where the emails do not match.
* Return a clear but non-enumerating error.
* Do not disclose sensitive invitation details.

Continue enforcing:

* Cryptographically secure random invitation tokens
* Expiration
* Single use
* Revocation
* Duplicate invitation prevention
* Token removal or invalidation after successful acceptance

# PART 2 — OWNER AND TENANT RELATIONSHIP RULES

## 6. One owner per apartment in the MVP

For the current MVP, an apartment may have a maximum of:

* One pending or active owner relationship

The data model may remain capable of supporting joint ownership later, but the current UI and backend must not allow a second owner.

Add PostgreSQL-level enforcement using a partial unique index or an equally reliable database constraint.

Example intention:

UNIQUE unit_id
WHERE role = 'owner'
AND status IN ('pending', 'active')

Keep the API pre-check for a user-friendly validation message.

Catch database unique-constraint violations and return HTTP 409 with a clear response.

---

## 7. One tenant per apartment in the MVP

An apartment may have a maximum of:

* One pending or active tenant relationship

Add PostgreSQL-level enforcement using a partial unique index or equivalent.

Example intention:

UNIQUE unit_id
WHERE role = 'tenant'
AND status IN ('pending', 'active')

Do not automatically revoke an existing tenant when another tenant is invited.

The owner or administrator must first:

* Cancel the pending invitation; or
* Revoke the current tenancy.

Catch database unique-constraint violations and return HTTP 409.

---

## 8. Relationship-specific revocation

Revoking one tenancy must revoke only that specific tenancy relationship.

It must not:

* Delete the user account.
* Revoke ownership of another apartment.
* Revoke another tenancy.
* Revoke administrator access.
* Remove unrelated company relationships.

One user may own multiple apartments and potentially hold different relationships across different companies.

## 12. Pricing effective dates

Each pricing configuration must have an effective-from date.

When calculating a billing month:

* Reject overlapping active configuration ranges or define deterministic priority rules.

A future pricing configuration must not affect current or previous billing months before its effective date.

* Enterprise pricing behaviour
* Currency

Once a billing month is finalised:

* Its thresholds must not change.
* Its rate must not change.
* Its tier must not be recalculated using new pricing.
* Its final amount must not change automatically.

Editing or adding a new live pricing configuration must never alter historical finalised records.

## 15. Billable property types

Only active records where:

unit_type = 'apartment'

currently count toward platform subscription billing.

Do not count:

* Garages
* Commercial units
* Common areas
* Other units

These non-apartment units may still exist in the property-management system, but they must not currently affect:

* Current billable apartment count
* Monthly peak apartment count
* Subscription tier
* Estimated monthly charge

Keep the architecture capable of making billable unit types configurable later, but do not build a complex interface for this now.

All user-facing billing wording must use “apartments”, not “units”.

## 16. Monthly peak billing rule

Billing is based on the highest number of simultaneously active billable apartments reached during the calendar month.

The applicable thresholds and rate must come from the pricing snapshot for that month.

Example using the seeded configuration:

* Peak 0–2: Free
* Peak 3–49: Standard
* Peak 50 or more: Enterprise/custom

Once Standard is reached, the per-apartment rate applies to every billable apartment.

Using the initial seeded configuration:

* Peak 3 × €5 = €15
* Peak 10 × €5 = €50
* Peak 49 × €5 = €245

These figures are examples based on the seeded configuration, not permanent business rules.

When an apartment is archived:

* Current billable apartment count decreases.
* Monthly peak does not decrease.
* Applicable tier for the month does not decrease.
* Estimated amount based on the peak does not decrease.

When an apartment is restored:

* Recalculate current billable apartment count.
* Increase the monthly peak only when the new active count exceeds the existing peak.

Owner invitation acceptance and tenant invitation acceptance must not affect billing.

---

## 17. Single source of truth for the current tier

Fix the disagreement between:

* companies.subscriptionTier
* monthly_usage_records.subscriptionTier

The current billing month’s tier must be based on the monthly peak and the applicable pricing snapshot.

Do not determine the current billing tier from the current active apartment count.

Prefer one reliable source of truth.

Recommended approach:

* Treat the current monthly usage record as the billing source of truth.
* Derive displayed plan/tier information from that record.
* Avoid maintaining a second independently calculated company-level tier.

When an apartment is archived after reaching Standard or Enterprise during the month, the current-month tier must remain based on the peak.

---

## 18. Atomic billing updates

Make current apartment-count and monthly-peak updates atomic.

Use:

* A PostgreSQL transaction with row locking; or
* An atomic update using GREATEST(); or
* An equally safe mechanism.

Concurrent apartment creation, restoration or archival must not:

* Under-count the monthly peak
* Produce a stale current apartment count
* Overwrite another operation’s calculation
* Produce an incorrect tier
* Produce an incorrect estimated amount

Test concurrent operations.

---

## 19. Month rollover and finalisation

Use Europe/Malta as the billing timezone.

When a new calendar month begins:

1. Identify the previous open monthly usage record.
2. Finalise it.
3. Set final_amount_cents.
4. Set invoice_status = finalised.
5. Prevent further automatic recalculation of that finalised record.
6. Select the pricing configuration applicable to the new billing month.
7. Apply any valid company-specific override.
8. Create the new monthly usage record.
9. Set current apartment count to the number of currently active billable apartments.
10. Set the starting peak equal to that current count.
11. Store the full pricing snapshot.

Use a scheduled job where available, plus a safe lazy first-request check as a fallback.

The rollover process must be idempotent.

Running it twice must not create duplicate records or alter a finalised month incorrectly.
